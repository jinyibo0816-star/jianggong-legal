# /matter-intake — 建工纠纷案件立卷

## 用途

为一个新接到的**建设工程施工合同纠纷**案件建立内部档案。所有后续 skill（起诉状起草、时效核查、时间线）都依赖此处生成的 `matter.md`。

## 工作原则

**一案一项目**：每个案件对应一个 Cowork 项目，项目文件夹既存客户原始资料、也存所有工作产出。因此本 skill 的所有输出**全部落在当前项目根目录**（用 `./` 表示），无需子文件夹分隔。

典型项目文件夹结构（执行完本 skill 后）：

```
<案件项目根>/
├── 客户原始资料/         （客户给你的合同、邮件、证据等，可能已存在）
├── matter.md             （本 skill 产出——案件主档）
├── _meta.yaml            （结构化索引）
└── _history.md           （操作日志）
```

## 执行步骤

### 第 1 步：确认案件标识

- 默认**案件简称 = 当前项目文件夹名**（用 `pwd` 或 `basename "$PWD"` 获取）
- 若 argument-hint 给了名字，用那个
- 让用户确认或修改
- 检查项目根目录下是否**已存在 `matter.md`**：
  - 存在 → 提示用户："本项目已有 matter.md，要覆盖重写、还是只更新缺失字段、还是取消？"
  - 不存在 → 继续

### 第 2 步：身份与角色

- **我方角色**：原告 / 被告 / 第三人 / 申请人 / 被申请人
- **当事人身份链**（建工特有，必问）：
  - 发包人（业主/建设单位）
  - 总承包人
  - 分包人 / 实际施工人
  - 出借资质企业（如涉及挂靠）
  - 我方在上述哪个位置？

### 第 3 步：当事人主体信息（按结构问）

对每个当事人（原告、被告各一组）按以下结构录入，未知字段留空：

```
名称：
住所地：
统一社会信用代码：
法定代表人/负责人：
联系电话：
所有制性质：国有控股 / 国有参股 / 民营 / 其他
类型：（公司/合伙企业/个体/自然人）
```

> 自然人当事人单独问：姓名、身份证号、住址、联系电话。

### 第 4 步：合同与施工事实

- 合同名称（如"东宇大厦施工总承包合同"）
- 签订日期
- 合同价款（元）
- 工程范围/承包范围
- 开工日期 / 计划竣工日期 / 实际竣工日期
- 是否已通过验收（合格/不合格/未验收）
- 是否办理结算（已办/未办/争议中）
- 结算金额（如有）
- 已付金额 / 欠付金额

> 如果当前项目文件夹下有客户原始资料（如 PDF/.docx 合同文件），可主动询问用户："要不要我先读一下你提供的合同文件、把能识别的字段自动填进来？"——用 Read 工具读取后填充，仍逐项给用户确认。

### 第 5 步：争议焦点初判

让用户用 1-3 句描述核心争议，并从下列**建工典型争点**中勾选（多选）：

- [ ] 工程价款数额（造价/工程量争议）
- [ ] 工期延误责任
- [ ] 工程质量缺陷
- [ ] 签证/变更/索赔效力
- [ ] 合同效力（无资质/超资质/转包/违法分包/挂靠）
- [ ] 优先受偿权主张
- [ ] 发包人欠付工程款连带责任
- [ ] 农民工工资专项
- [ ] 保函/保证金返还
- [ ] 鉴定相关
- [ ] 其他：____

### 第 6 步：关键时间节点（先粗后细）

至少录入以下能确定的节点，**只问用户能立刻给出的**，剩余留待 `/chronology` 详细梳理：

- 合同签订日
- 开工日
- 停工日（如有）
- 竣工日（实际）
- 验收日（如有）
- 结算完成日（如有）
- 最后一次催告日（关键——影响时效）
- 起诉/拟起诉日

### 第 7 步：代理人安排

调用 MCP 工具 `list_lawyer_profiles` 列出已沉淀的律师档案，让用户选 1-2 位作为代理人。如未在档案中，提示是否现在用 `save_lawyer_profile` 新建。

### 第 8 步：生成案件档案

在**当前项目根目录**（`./`）创建以下三个文件：

#### `./matter.md`（主档）

```markdown
# <案件简称>

- **立卷日期**: <今天日期>
- **我方角色**: <原告/被告>
- **承办律师**: <代理人姓名>
- **案件阶段**: 立案前 / 诉讼中 / 二审 / 执行

## 当事人

### 原告
名称：
统一社会信用代码：
法定代表人：
联系电话：
住所地：

### 被告
（同上结构）

## 主体身份链
- 发包人：
- 总承包人：
- 分包人 / 实际施工人：
- 出借资质企业：
- 我方位置：

## 合同与施工事实
- 合同名称：
- 签订日期：
- 合同价款：
- 开/竣工：
- 验收情况：
- 结算情况：
- 已付 / 欠付：

## 争议焦点
（用户给的 1-3 句概述 + 勾选的典型争点）

## 关键时间节点
（用户给出的节点列表）

## 备注
（用户补充的其他信息）
```

#### `./_meta.yaml`（结构化索引）

```yaml
name: <案件简称>
created: <YYYY-MM-DD>
role: 原告/被告
stage: 立案前
counterparty: <对方名称>
contract_amount: <数字>
disputed_amount: <数字>
typical_issues:
  - 工程价款数额
  - ...
lawyers:
  - 李中
  - 靳艺博
```

#### `./_history.md`（操作日志）

```markdown
# 操作日志

- <YYYY-MM-DD HH:MM> 案件立卷 — matter.md 初版生成
```

### 第 9 步：确认与下一步

向用户输出：
- "案件已立卷到当前项目根目录"
- 列出已记录的主要字段
- 提示：「下一步可以跑 `/chronology` 详细梳理时间线，或 `/pleading-draft` 起草起诉状，或 `/statute-of-limitations` 核查诉讼时效。」

## 重要约束

- **不要瞎填**：任何用户没明确给出的字段都留空（空字符串），不要根据上下文猜测当事人名称、金额、日期。
- **所有产出落在 `./`**（当前项目根），不要写到任何其他绝对路径。
- **如果你不确定当前 cwd 在哪里**，先 `pwd` 确认；若 cwd 看起来是 `/开发/` 这种通用开发目录而不是某个具体案件文件夹，提醒用户："看起来你在 legal-kb 的开发目录而不是案件项目里，是否要继续在这里立卷？"
- **建工身份链是必问项**：发包人/总包/实际施工人/出借资质的关系决定后续诉讼路径，不能跳过。
- **金额单位统一为"元"**，不要用"万元"混写。
- **日期统一 YYYY-MM-DD 格式**。
- 当事人是自然人还是法人要明确——影响 `要素式起诉状` 的填充逻辑。
- 任何输出都是律师审核前的草稿。不要在 `matter.md` 里写任何形如"建议起诉"、"主张胜算大"的判断——只记录事实，不下结论。
