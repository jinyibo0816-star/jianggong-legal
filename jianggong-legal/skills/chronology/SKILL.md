# /chronology — 建工案件时间线梳理

## 用途

建工纠纷案件的事实判断高度依赖时间链。这个 skill 把零散的合同、签证、催告、停工通知等事件按时间排列，生成一份**带证据来源**的标准时间线，并自动识别风险节点（如停工无催告、变更无签证、超期未结算、临近时效届满等）。

## 工作原则

**一案一项目**——当前 Cowork 项目根目录就是这个案件的工作目录。

- 输入：读 `./matter.md`（提取已立卷时录入的粗节点）
- 输出：写 `./chronology.md`（可读版）+ `./chronology.yaml`（结构化）
- 操作日志追加到 `./_history.md`

## 建工标准节点清单（按时间顺序）

按下列结构引导用户录入。**不存在的节点跳过，不要硬填**。

### 1. 招投标阶段（如适用）
- 中标通知书发出日
- 中标公示期满日
- 招标文件 / 投标文件签字日

### 2. 合同签订阶段
- **施工合同签订日**（基础节点）
- 补充协议签订日（每份单独列）
- 专业分包合同签订日
- 材料/设备采购合同签订日
- 履约保函/履约保证金交付日

### 3. 开工阶段
- 开工报告 / 开工令日期
- 实际进场日
- 设计图纸交付日
- 监理通知日

### 4. 施工过程节点（每个独立事件单列）
- 工程变更指令日 + 变更内容摘要
- 签证日 + 签证编号 + 签证内容摘要 + 签证金额
- 索赔通知日 + 索赔事项
- 停工通知日 + 停工原因 + 停工责任主体
- 复工通知日
- 监理周报/月报关键节点（如适用）
- 进度款支付节点（每次实际付款日 + 金额）

### 5. 竣工阶段
- 承包人竣工报告提交日
- 实际完工日（封顶/最后施工日）
- 竣工验收申请日
- **竣工验收合格日**（关键节点——影响保修期与时效）
- 竣工验收备案日
- 实际交付/接管日

### 6. 结算阶段
- 承包人结算资料提交日
- 发包人审核回函日（如有）
- 双方签字确认结算日 / 结算金额
- 工程款应付日（合同约定 or 结算后约定）
- 实际最后付款日 + 金额

### 7. 催收与争议阶段
- 每次催告日 + 形式（书面/电子/口头）+ 证据形式 + 收件人
- 对方回函/承认日（影响时效中断）
- 协商记录关键节点
- 调解 / 仲裁 / 监管部门投诉日
- **保函索赔日**（如有）

### 8. 诉讼准备阶段
- 律师介入日
- 调查取证关键节点
- 诉前财产保全申请日 + 法院 + 案号 + 裁定日
- 起诉日（已起诉）或拟起诉日

## 执行步骤

### 第 1 步：定位案件 & 读取已有信息

- `pwd` 确认当前在案件项目目录
- 读 `./matter.md`（不存在则提示用户先跑 `/matter-intake`），提取已有时间节点（matter-intake 第 6 步录入的粗节点）
- 若已有 `./chronology.md`，进入"更新模式"——只补新节点，不重复问

### 第 2 步：分阶段引导录入

按上述 8 个阶段，**逐段问**用户。每个事件至少录入：

```yaml
- date: YYYY-MM-DD
  stage: 施工过程   # 招投标/合同/开工/施工/竣工/结算/催收/诉讼
  event: 第3次签证   # 简短描述
  detail: |
    XX 部位变更增加 X 元工程量；监理 XX 签字确认。
  evidence: 签证单 编号 BG-2024-003 / 监理报告页 P12-13
  parties: [总包-中建X局, 监理-XX]
  amount: 285000   # 涉及金额（元），无则省略
  risk_flag: ""   # 仅在自动识别到风险时填写
```

每段录入完，向用户确认"是否还有 [本阶段] 其他节点"，再进入下一阶段。

### 第 3 步：自动识别风险节点

录完后自动扫描，对以下情形打 `risk_flag`：

| 风险类型             | 触发条件                                                | 标记内容                                  |
|----------------------|---------------------------------------------------------|-------------------------------------------|
| 停工未催告           | 停工事件后 30 日内无书面催告 / 索赔                     | ⚠️ 停工责任分担存疑                       |
| 变更无书面签证       | 录入了"变更指令"但同期无"签证单"证据                    | ⚠️ 变更价款追加可能不被支持               |
| 竣工后超期未结算     | 实际完工日 > 90 日，仍无结算节点                        | ⚠️ 可能触发"视为认可"或工程款主张时效起算 |
| 优先受偿权临界       | 应当给付价款日 + 18 个月与今日距离 < 90 日              | 🔴 优先受偿权除斥期间即将届满            |
| 普通时效临界         | 最后中断日 + 3 年与今日距离 < 90 日                    | 🔴 诉讼时效即将届满                       |
| 催告证据形式弱       | 催告事件 evidence 字段中只有"口头"/"电话"               | ⚠️ 时效中断可能不成立                     |
| 结算文件未签字       | 结算事件 detail 中未明确"双方签字"                      | ⚠️ 结算金额可能被对方推翻                 |

### 第 4 步：生成两份输出（都在当前项目根目录）

#### `./chronology.yaml`（结构化）

按时间排序的完整事件 list（结构见第 2 步）。

#### `./chronology.md`（可读版）

```markdown
# <案件简称> 时间线

生成日期：<YYYY-MM-DD>

## 风险节点提示

🔴 **紧急**
- YYYY-MM-DD —— 优先受偿权 18 个月行使期届满（剩余 X 天）

⚠️ **关注**
- 停工事件（YYYY-MM-DD）后 30 日内未见书面索赔记录

## 时间线（按阶段）

### 1️⃣ 合同签订阶段

| 日期       | 事件             | 涉及金额 | 证据来源              | 备注 |
|------------|------------------|----------|-----------------------|------|
| 2023-03-15 | 施工总承包合同签订 | 1.2 亿元 | 合同原件 / 印鉴页 P1  |      |
| 2023-04-02 | 补充协议（一）   | -        | 补充协议原件         |      |

### 2️⃣ 开工阶段
（表格同上）

…（其他阶段同结构）

## 缺口提示

以下建工标准节点本案未录入，请确认是否存在：
- [ ] 竣工验收备案日
- [ ] 履约保函交付日
- [ ] 监理周报关键节点

## 引用证据清单（去重）

- 合同原件
- 补充协议（一）
- 签证单 BG-2024-001 至 BG-2024-008
- ...
```

### 第 5 步：交付

- 告知用户两份文件路径（`./chronology.md` + `./chronology.yaml`）
- 列出 🔴 紧急项数量与 ⚠️ 关注项数量
- 在 `./_history.md` 追加一行：`- <YYYY-MM-DD HH:MM> 时间线梳理完成 → chronology.md / chronology.yaml`
- 建议下一步：若有 🔴 紧急项 → 跑 `/statute-of-limitations` 详细核查；若准备起诉 → 跑 `/pleading-draft`

## 重要约束

- **所有产出落在 `./`**（当前项目根），不要写到任何其他绝对路径。
- **每个节点必须有证据来源**——`evidence` 字段不能为空字符串。若用户说"记得有但找不到证据"，记为 `evidence: 待补充原始证据`，并打 ⚠️ flag。
- **金额单位统一"元"**，不要用"万"或"亿"混写。如用户说"1.2 亿"，转写为 `120000000`。
- **日期不能模糊**——"大概 2023 年中"这种要追问用户回到原始证据中确认。如确实不可得，记为 `date: ~2023-06`（带 `~` 前缀表示估算），并打 ⚠️ flag。
- **不要自己造节点**——只录用户明确给出或现有材料中能定位的节点。
- 涉及多个被告 / 多份合同的复杂建工案件，可以按 `chain: 主合同 / 分包合同 A / 分包合同 B` 字段分链路标记，避免混线。
- 任何输出都是律师审核前的草稿。风险标记是机械规则触发，不是法律判断——最终由律师定。
